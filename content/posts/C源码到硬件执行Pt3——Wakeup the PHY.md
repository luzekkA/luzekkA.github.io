## 第一部分：程序真正的入口

在Pt3之前，我们已经成功构建了一幅逻辑内存模型的全景图，但是如果需要让我们的机器成功执行这个程序，我们必须完成从“逻辑”到“物理”的映射：我们需要思考如何把这些逻辑内存里面的东西装载到物理内存里面，然后让CPU上电执行，我们需要根据不同数据段的特性把程序放到不同的RAM，ROM中。

这便引出了链接脚本的两个核心使命：**描述程序在物理空间中的布局**，以及**定义程序执行的起点**。

当然真实脚本复杂得多，但核心结构就是这样。

我们可以先来看看这三个顶层指令->

##### OUTPUT_FORMAT

```
OUTPUT_FORMAT("elf64-x86-64")
```

告诉链接器：

> 生成什么格式的 ELF

这和目标架构绑定。

##### OUTPUT_ARCH

```
OUTPUT_ARCH(i386:x86-64)
```

指定 CPU 架构。

##### ENTRY(_start)

**告诉链接器：程序的执行起点在哪里。**

- **符号绑定**：它指定 `_start` 这个符号作为整个程序的入口地址。
- **写入文件头**：链接器会将 `_start` 对应的 **VMA（运行地址）** 写入 ELF 文件头的 `e_entry` 字段中。
- **启动逻辑**：
  - **在 Linux 中**：当你运行 `./a.out` 时，内核加载器会读取这个 `e_entry` 地址，并将 CPU 的指令指针（PC/RIP）跳转到那里开始执行。
  - **在裸机中**：虽然硬件上电通常跳转到固定的物理地址，但 `ENTRY` 指令依然不可或缺，它能让调试器（如 GDB）准确知道程序从哪开始，同时也方便链接器在进行垃圾回收（--gc-sections）时，知道 `_start` 是“根符号”，不应被删除。

### ENTRY指令

#### ENTRY(_start) 到底在干什么？

在链接脚本里：

```
ENTRY(_start)
```

它的作用只有一句话：

> 把 ELF Header 里的 `e_entry` 字段设置为符号 `_start` 的地址。

也就是说：

- 它不生成代码
- 它不改变 section 布局
- 它不影响 reset 向量
- 它只是指定“程序入口符号”

你可以验证：

```
readelf -h hello
```

会看到：

```
Entry point address: 0xXXXX
```

这个地址就是 `_start` 的 VMA。关于什么是VMA和LMA我们在下文会又比较详细的描述，在这里你可以理解为程序在内存的地址。

------

#### 那 `_start` 是什么？

无论是在 Linux 还是裸机（Bare-metal）中，`_start` 的任务都是为了给 `main` 铺路。它通常是一段汇编代码，干的事情非常原始：

##### 在 Linux 环境下：

1. **从栈上获取信息**：当 Linux 内核启动程序后，它会把 `argc`、`argv` 和环境变量压入栈。`_start` 负责把这些数据从栈里掏出来。
2. **初始化 C 库**：调用 `__libc_csu_init` 之类的逻辑，初始化全局构造函数（比如那些带 `__attribute__((constructor))` 的函数）。
3. **把球踢给 `main`**：调用 `__libc_start_main`，这才是真正调 `main` 的地方。
4. **清理现场**：当 `main` 返回后，`_start` 会调用 `exit` 系统调用来彻底杀死进程。

##### 在裸机环境下（RISC-V 为例）：

1. **设置 SP 指针**：`la sp, _stack_top`。这是第一件大事，有了栈，C 代码才能运行。
2. **全局指针初始化**：设置 `gp` 寄存器（RISC-V 特有），加速全局变量访问。
3. **搬运与清零**：
   - 把 `.data` 段从 Flash 拷贝到 RAM。
   - 把 `.bss` 段全部抹成 0。
4. **跳转到 main**：`call main`。

#### 在 Linux 中 `_start` 从哪来？

在Pt1中我们只用gcc将程序编译成可执行文件：

```
gcc hello.c -o hello
```

即使我们没有告诉他 `_start` 的内容，程序仍然可以运行。

因为 GCC 自动链接了：

- `crt1.o`
- `crti.o`
- `crtn.o`

其中在 `crt1.o` 里定义了：

```
_start:
    ...
    call __libc_start_main
```

然后：

```
__libc_start_main → 调用 main
```

所以执行流程是：

```
_start → libc 初始化 → main → exit
```

------

#### 在裸机中 `_start` 是谁？

在裸机环境里：

- 没有 libc
- 没有 crt1.o
- 没有 loader

所以你必须自己写 `_start`

通常是：

```
.section entry
.global _start

_start:
    la sp, _stack_top
    call init_data
    call zero_bss
    call main
loop:
    j loop
```

#### ENTRY() 还能写别的吗？

理论上是可以的。

```
ENTRY(reset_handler)
```

或者：

```
ENTRY(main)
```

只要这个符号存在即可。

链接器只做一件事：找到这个符号的地址 → 写入 ELF header

#### 如果不写 ENTRY() 会怎样？

ld 有默认规则：

优先级是：

1. 命令行 `-e symbol`
2. 链接脚本 `ENTRY(symbol)`
3. 查找 `_start`
4. 查找 `start`
5. 取 `.text` 开头

所以即使你不写 ENTRY：

如果存在 `_start`，通常仍然会选它。

#### ENTRY ≠ 硬件复位地址

这点非常关键（尤其是对于裸机环境）。

在 Linux：

- 内核读取 ELF header
- 跳转到 `e_entry`

在裸机：

比如CPU 上电后：

- PC 从固定物理地址开始（例如 0x20000000）
- 这与 ELF header 无关

所以在裸机中：

如果你把代码烧到 0x20000000
 那 `_start` 必须刚好位于这个地址

否则 ENTRY 没意义。

## 第二部分：双面龟 —— LMA vs VMA

### 什么是VMA与LMA？

在 ELF 中，每个输出 section 都有两个地址：

| 名称                         | 含义                   |
| ---------------------------- | ---------------------- |
| VMA (Virtual Memory Address) | 程序运行时访问的地址   |
| LMA (Load Memory Address)    | 程序被加载时存放的地址 |

默认情况下：

```
VMA == LMA
```

只有在嵌入式场景下才会分离

我们可以来看一个简单的脚本片段

```
MEMORY
{
  MROM (rx)  : ORIGIN = 0x20000000, LENGTH = 128K
  SRAM (rwx) : ORIGIN = 0x0f000000, LENGTH = 8K
}
```

这一步做的事情是：

- 定义两个可分配内存区域
- 后续会通过 `> MROM` 或 `> SRAM` 来引用它们

### 为什么我们需要LMA与VMA

要回答这个问题，我们必须跳出 C 语言的逻辑世界，回到**物理硬件的约束**中去。

简单来说，我们需要 LMA 和 VMA 分离，是因为**有些数据装载时的地址和运行时的地址不是同一个地址。**

#### Flash 只能读，RAM 才能写的物理矛盾

这是最根本的原因。请看以下这个最简单的全局变量：

```
int count = 100; // 已初始化数据（.data）
```

- **断电不能丢（存储需求）**：这个 `100` 必须烧录在 **Flash** 里。如果它只存在 RAM 里，一断电，你的程序下次启动就不知道 `count` 初始值是 100 了。
- **运行时要改（执行需求）**：程序运行到 `count++` 时，CPU 必须能写这个内存。但 Flash（ROM）是只读的，CPU 无法直接通过一条指令修改 Flash 里的值。

**矛盾点出现了：** 这个变量在镜像文件中必须位于 Flash 地址（**LMA**），但在程序运行时，它必须位于 RAM 地址（**VMA**）。

#### LMA 与 VMA 的分工

为了解决这个矛盾，链接器给 `.data` 段设置了两套地址：

- **LMA (Load Memory Address - 加载地址)**： 这是程序**“躺”在 Flash 里的地址**。它决定了烧录器把数据写到芯片的哪个物理位置。它负责**数据的持久化存储**。
- **VMA (Virtual/Runtime Memory Address - 运行地址)**： 这是程序**“跑”在 RAM 里的地址**。编译器生成的指令（比如 `load` 或 `store`）里用的就是这个地址。它负责**数据的动态读写**。

#### 裸机中的典型场景

假设你的单片机：

- **Flash (MROM)** 范围：`0x20000000` 开始
- **RAM (SRAM)** 范围：`0x0F000000` 开始

如果你在链接脚本里强行让 `.data` 的 LMA 等于 VMA（都放在 Flash 里），那么当你执行 `count++` 时，CPU 会尝试往 `0x2000xxxx` 写数据，硬件会直接报错（Bus Fault）。

**正确的做法是分离：**

1. **链接阶段**：告诉链接器，`.data` 的数据在二进制文件里跟在代码后面（LMA 在 Flash），但指令寻址时要去 RAM 找（VMA 在 RAM）。
2. **启动阶段**：这就是你在 `_start` 里要做的事——把那段数据从 Flash 物理地搬运到 RAM 中。

#### 为什么 `.text` 段通常不需要分离？

代码段（`.text`）和常量（`.rodata`）通常 LMA == VMA。

因为它们是**只读**的。现代 CPU 普遍支持 **XIP (Execute In Place)**，即直接从 Flash 读取指令并执行。既然不需要写，也就不需要挪地方，直接让它“住在 Flash，工作在 Flash”即可。

所以如果没有 LMA/VMA 的分离机制：

1. **全局变量将无法修改**（因为它们只能死在 Flash 里）。
2. **或者所有数据必须全部预先加载进 RAM**（但 RAM 非常昂贵且容量有限，存不下庞大的代码段）。

**LMA 和 VMA 的存在，本质上是链接器为了在“有限且昂贵的 RAM”与“大容量且廉价的 Flash”之间寻找平衡的一种地址映射手段。**

## 第三部分：一些简单的ld语法

### 符号定义

在链接脚本里，你可以像写 C 语言赋值语句一样定义符号：

```
_stack_top = 0x2000FFFF;
```

**注意**：这只是定义了一个**符号地址**，它不占用任何 Flash 空间。

**C 语言如何调用**：你在 C 代码里写 `extern int _stack_top;`，那么 `&_stack_top` 的值就是 `0x2000FFFF`。

### 定位计数器 `.` (The Location Counter)

这是链接脚本中最灵魂的符号。它代表了**当前的虚拟地址（VMA）**。

**特性**：链接器按顺序处理段（Section），处理完一个，`.` 就会自动往后移动相应的大小。

**手动修改**：你可以通过 `. = 0x10000;` 强行把地址推到某个特定位置，常用于**对齐（Alignment）**。

### MEMORY 指令

在定义具体的段之前，我们要先告诉链接器：这块芯片到底有多少物理内存。

代码段

```
MEMORY
{
  MROM (rx)  : ORIGIN = 0x20000000, LENGTH = 1M
  SRAM (rwx) : ORIGIN = 0x0F000000, LENGTH = 64K
}
```

**r (read)**, **w (write)**, **x (execute)**：这是给内存打的权限标签。

有了这个，后面我们才能用 `> SRAM` 这种简洁的语法把段扔进指定的物理区间。

### SECTIONS 基础

这是链接脚本的大头，负责决定哪些输入文件（.o）的哪些段放在输出文件的哪个位置。

```
SECTIONS
{
  .text : 
  { 
    *(.text)    /* 收集所有输入文件的 .text 段 */
    *(.rodata)  /* 收集所有输入文件的只读数据 */
  } > MROM      /* 放入 MROM 区域 */
}
```

**通配符 `\*`**：代表“所有文件”。

**括号内**：是输入段（Input Section）；**括号外**：是输出段（Output Section）。

### PROVIDE 关键字

```
PROVIDE(end = .);
```

它的意思是：如果你的 C 代码里没定义 `end` 这个符号，那就用链接脚本里的这个值；如果你自己定义了，那就以你定义的为准。这是标准库（如 `malloc` 依赖的 `_end`）常用的技巧。

### KEEP 关键字

```
KEEP(*(.vectors))
```

在开启了链接器优化（`--gc-sections`）时，没有被调用的代码会被删掉。但像**中断向量表**这种硬件自动调用的代码，编译器看不出来它被调用了。加上 `KEEP` 就能防止它被误删。

## 第四部分：AT语法

`AT()` 指令是专门用来指定 **LMA（加载地址）** 的。

在默认情况下，如果你写 `.data : { ... } > SRAM`，链接器会认为这个段的 VMA 和 LMA 都在 SRAM 里。但这会导致一个严重问题：**烧录器找不到数据该往哪存，因为 SRAM 是掉电即失的，它不能存放二进制镜像。**

于是，我们祭出了这段核心代码：

代码段

```
. = ORIGIN(SRAM); /* 强制将当前定位符设为 SRAM 起始地址 0x0f000000 */

.data : AT(LOADADDR(.rodata) + SIZEOF(.rodata))
{
  _data_start = .; 
  *(.data*)
  _data_end = .;
} > SRAM
```

这里的逻辑：

1. **确定运行地址 (`> SRAM`)**：告诉链接器，`.data` 里的变量在程序跑起来后，去 `0x0f000000` 这个位置找。
2. **标记符号地址 (`_data_start = .`)**：此时的 `.` 已经指向了 SRAM，所以这个符号记录的是 **VMA**。
3. **指定存储位置 (`AT(...)`)**：这是最核心的计算。
   - `LOADADDR(.rodata)`：拿到只读数据段在 ROM 里的起始加载地址。
   - `SIZEOF(.rodata)`：拿到只读数据段的大小。
   - 两者相加，意思就是：**“请把 `.data` 的初始值，紧紧贴在 ROM 里 `.rodata` 的屁股后面存着。”**

### 在 ELF 文件里，这如何体现？

我们可以通过 `readelf -l` 命令观察编译出的 ELF 文件，你会发现 Program Header 里出现了有趣的现象：

Plaintext

```
Program Headers:
  Type           Offset             VirtAddr           PhysAddr
  LOAD           0x0000000000001000 0x20000000         0x20000000  /* .text & .rodata */
  LOAD           0x0000000000002000 0x0f000000         0x20000500  /* .data */
```

- **VirtAddr (p_vaddr)**：这就是 **VMA**，程序运行时认为自己所在的地址。
- **PhysAddr (p_paddr)**：这就是 **LMA**，烧录器真正把数据写进去的地址。
- **结论**：在嵌入式中，`p_paddr` 是烧录地址，`p_vaddr` 是运行地址。

### 为什么 `.bss` 没有 AT()？

观察你的链接脚本，`.bss` 通常写得很简单：

```
.bss :
{
  *(.bss*)
} > SRAM
```

**原因很简单：**.bss 段存放的是未初始化的全局变量（默认全为 0）。

- 它不需要初始值，所以**不需要在 ROM 里占用任何存储空间**。
- 既然 ROM 里没它，它自然就不需要 LMA。
- 它只需要在运行时（VMA）在 SRAM 里占个坑，然后由启动代码把它抹成零即可。

### Startup Code 的责任

链接脚本只是画好了“规划图”，它并不会自动搬运数据。如果你的 `_start` 汇编里没写搬运逻辑，程序一跑就会因为找不到变量初值而崩溃。

这就是为什么我们需要在 `_start` 里引用链接脚本导出的符号：

1. 使用 `_data_load_obj` (LMA起始) 找到 Flash 里的源头。
2. 使用 `_data_start` (VMA起始) 找到 SRAM 里的目的地。
3. 执行一趟 `memcpy`。

**`>` 决定 VMA（在哪跑），`AT()` 决定 LMA（在哪存）。**

如果不写 `AT()`，链接器默认 **LMA = VMA**。这在 Linux 开发中很常见（因为加载器会自动处理），但在裸机开发中，这往往意味着你的程序无法正常启动。

## 第五部分：裸机启动环境

#### 现在我们可以回到你的 ld 脚本

```
.text :
{
  *(entry)
  *(.text*)
} > MROM
```

这一句做了两件事：

1. 决定顺序（entry 先放）
2. 决定地址（从 0x20000000 开始）

这就是“分配地址”阶段。

```ld
MEMORY
{
  MROM (rx)  : ORIGIN = 0x20000000, LENGTH = 128K
  SRAM (rwx) : ORIGIN = 0x0f000000, LENGTH = 8K
}

ENTRY(_start)

SECTIONS
{
  . = ORIGIN(MROM);

  .text :
  {
    *(entry)
    *(.text*)
  } > MROM

  .rodata :
  {
    *(.rodata*)
  } > MROM

  . = ORIGIN(SRAM);

  .data : AT(LOADADDR(.rodata) + SIZEOF(.rodata))
  {
    _data_start = .;
    *(.data*)
    _data_end = .;
  } > SRAM

  _data_load = LOADADDR(.data);

  .bss :
  {
    _bss_start = .;
    *(.bss*)
    *(COMMON)
    _bss_end = .;
  } > SRAM

  . = ALIGN(8);
  _heap_start = .;

  _stack_top = ORIGIN(SRAM) + LENGTH(SRAM);
}

```

这个脚本就是一个典型的“ROM to RAM copy model”

内存布局是：

ROM:

```
.text
.rodata
.data (initial image)
```

SRAM:

```
.data (runtime)
.bss
.heap
.stack
```

现在你可以看到完整逻辑：

1. CPU 复位
2. SP 未知
3. 必须先设置栈
4. 才能安全调用函数
5. 再初始化 .data / .bss
6. 再进入 C 语言世界

堆是“附加能力”
 栈是“基础设施”

#### 一、启动代码（startup code）本质是什么？

一句话定义：

> 启动代码 = 在进入 C 语言世界之前，建立运行环境的那段最低级代码。

CPU 上电后不会自动帮你：

- 设置栈
- 初始化全局变量
- 清零未初始化变量
- 调用构造函数
- 跳转到 main()

这些都必须由你（或者 libc）完成。

------

#### 启动代码必须做的几件事（按顺序）

你目前讲的是偏裸机 / bootloader 场景，我重点讲这个。

------

#### 1️⃣ 必须先设置栈（Stack Pointer）

因为 C 语言依赖栈。

例如：

```
int main() {
    int a = 5;
}
```

变量 `a` 在栈上。

函数调用：

```
foo();
```

会：

- 把返回地址压栈
- 保存寄存器
- 分配局部变量空间

如果 SP 没有指向合法内存：

- push 会写到未知地址
- 函数调用直接崩溃
- 甚至访问到非法地址导致异常

所以启动代码第一步通常是：

```
ldr sp, =_stack_top
```

或

```
mov sp, #0x20010000
```

这一步建立运行时环境的“地基”。

------

#### 2️⃣ 为什么必须复制 .data 段？

这涉及 LMA / VMA。

典型裸机内存布局：

Flash：

```
.text
.rodata
.data  (初始化数据的存储副本)
```

RAM：

```
.data  (真正运行时的)
.bss
.stack
.heap
```

关键点：

.data 段：

- 运行地址在 RAM
- 初始值存储在 Flash

例如：

```
int g = 5;
```

编译器会把 5 放到 Flash。

但运行时：

```
g 必须在 RAM 中
```

因为：

- 全局变量可能被修改
- Flash 通常不可写

所以启动代码必须：

```
把 Flash 中的 .data 复制到 RAM 中的 .data
```

否则：

- g 读取值是错的
- 或者访问未初始化内存

典型代码：

```
uint32_t *src = &_sidata;   // LMA
uint32_t *dst = &_sdata;    // VMA
while (dst < &_edata) {
    *dst++ = *src++;
}
```

#### 为什么必须清零 .bss？

.bss 段：

存放未初始化的全局变量

```
int x;
```

按照 C 标准：

> 未初始化的全局变量默认必须为 0

但：

- Flash 里不会给它存 0
- RAM 上电后是随机值

所以必须清零。

启动代码必须做：

```
uint32_t *p = &_sbss;
while (p < &_ebss) {
    *p++ = 0;
}
```

否则：

程序逻辑会随机错误。

------

#### 最后才能调用 main()

完整流程：

```
CPU reset
  ↓
跳到 reset handler
  ↓
设置 SP
  ↓
复制 .data
  ↓
清零 .bss
  ↓
调用 C++ 构造函数（如果有）
  ↓
调用 main()
```

------

#### 如果不做这些会发生什么？

| 忽略步骤       | 后果                 |
| -------------- | -------------------- |
| 不设栈         | 函数调用直接崩溃     |
| 不复制 .data   | 初始化全局变量值错误 |
| 不清零 .bss    | 未初始化变量随机值   |
| 不调用构造函数 | C++ 静态对象出错     |


你没看到：

```
. = 0x400000
```

这不是你看漏了，而是因为：

> 你现在编译出来的是 PIE（Position Independent Executable）。

------

#### 为什么是 DYN 而不是 EXEC？

注意这一行：

```
类型: DYN (Position-Independent Executable file)
```

在现代 Linux（Ubuntu 默认 GCC）中：

```
gcc hello.c -o hello
```

默认开启：

```
-fPIE
-pie
```

所以生成的不是传统的：

```
EXEC
```

而是：

```
DYN
```

它和共享库使用同一种 ELF 类型。

------

#### PIE 意味着什么？

PIE 的含义是：

> 程序可以被加载到任意虚拟地址。

因此：

- 链接时不能固定 `. = 0x400000`
- 所有代码是相对地址
- 真正加载地址由内核随机决定（ASLR）

所以你看到：

```
入口点地址：0x1060
```

这其实是：

```
相对于加载基址的偏移
```

而不是绝对虚拟地址。

------

#### 那 0x400000 到底去哪了？

在老式（非 PIE）可执行文件中：

```
gcc hello.c -no-pie -o hello
```

然后再：

```
readelf -h hello
```

你会看到：

```
类型: EXEC
入口点地址: 0x401060
```

这时候：

- `.text` 从 0x400000 开始
- `_start` 在 0x401060

也就是说：

```
0x400000 + 0x1060 = 0x401060
```

而你现在的：

```
0x1060
```

是 PIE 模式下的“相对入口”。



所以在非PIE版本中所有程序的入口都在0x400000？那为什么两个不一样的程序不会进入到同一段代码呢

即使两个非-PIE 程序的入口都是 0x400000，它们也不会进入同一段代码，因为每个进程拥有独立的虚拟地址空间。



#### 虚拟地址空间是“每个进程独立的”

在 Linux 上：

- 每个进程都有自己的页表
- 每个进程的虚拟地址空间互相隔离
- 0x400000 在不同进程里是不同的映射

举个例子：

假设你有两个程序：

- hello
- world

它们都是 non-PIE，入口都是 0x401060。

当你同时运行：

```
./hello &
./world &
```

它们的 `/proc/<pid>/maps` 可能看起来像：

```
00400000-00401000 r-xp hello
00400000-00401000 r-xp world
```

看起来地址一样。

但：

- hello 的 0x400000 映射到物理页 A
- world 的 0x400000 映射到物理页 B

因为它们的页表不同。
