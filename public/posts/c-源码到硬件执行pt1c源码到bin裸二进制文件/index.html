第一部分：gcc 做了什么#

gcc hello.c -o hello 到底做了什么？#

我们从所有人都熟悉的一条命令开始：

gcc hello.c -o hello

很多人以为 gcc 做了一件事：把 C 变成可执行文件。

实际上它做了四件完全不同的事情。

1.预处理（Preprocessing）# 

gcc -E hello.c

发生的事情：

  • 展开 #include
  • 展开宏
  • 条件编译处理
  • 删除注释

输出结果是：巨大的纯 C 文件

此时：

  • 没有编译
  • 没有生成机器码
  • 只是文本替换

2.编译（Compilation）# 

gcc -S hello.c

发生的事情：

  • 语法分析
  • 生成抽象语法树
  • 语义检查
  • 优化
  • 生成汇编代码

输出：

main:
    addi sp, sp, -16
    ...

此时：

  • 仍然没有机器码
  • 只是目标架构的汇编

3.汇编（Assembly）# 

gcc -c hello.c

这一步会：

  • 把汇编变成机器码
  • 生成 ELF 格式的可重定位目标文件,关于可重定位目标文件，我们会在下文进行展开描述。

输出：

hello.o

这时的文件是：

readelf -h hello.o

会显示：

Type: REL (Relocatable file)

关键点来了：.o 文件不能运行。

因为：

  • 所有函数地址还未确定
  • 外部符号未解析
  • section （代码节 如.text 我们在下文会有详细介绍） 只是逻辑片段
  • 没有最终内存布局

关于.o文件为什么不能文件，将在下文有更详细的讲述。

4.链接（Linking）#

这才是：

gcc hello.c -o hello

的最后一步。

等价于：

gcc hello.o -lc -o hello

然后 gcc 会自动调用：

ld

并且：

  • 链接 libc
  • 链接启动文件
  • 选择默认链接脚本
  • 分配虚拟地址
  • 生成最终 ELF

现在文件类型变成：

readelf -h hello
Type: EXEC (Executable file)
链接器如何把多个 .o 变成一个可执行文件？#

我们可以用一个非常简单的例子：

main.c

int foo();
int main() {
  return foo();
}

foo.c

int foo() {
  return 42;
}

然后：

gcc -c main.c
gcc -c foo.c
gcc main.o foo.o -o hello

现在问题来了：

关键问题

在 main.o 里：

  • foo 是未定义符号
  • 地址未知

在 foo.o 里：

  • foo 有定义
  • 但地址仍然未知

链接器的具体行为过程如下

  1. 收集所有 .o 的符号表
  2. 建立全局符号表
  3. 找到：
    • main 里需要 foo
    • foo.o 提供 foo
  4. 决定：
    • foo 最终地址是多少
    • main 调 foo 的跳转要修正为多少

这个过程叫：符号解析 + 重定位

第二部分：.o 文件是什么#

.o文件#

简单来说，.o 文件（Object File，目标文件）是 ELF 格式的一种具体实现形态。在 Linux 和大多数嵌入式系统中，.o 文件本质上就是 ELF 格式。ELF 是一种统一的文件格式规范，而 .o 文件（可重定位文件）是该规范在编译流程中间阶段的具体呈现形式，其目的是为后续的链接过程提供标准化的代码与数据输入。

以下是它们关系的详细教科书式定义：

  1. 定义与归属关系

  • ELF (Executable and Linkable Format)：是一种文件格式标准。它定义了二进制文件（代码、数据、符号等）在硬盘上应该如何组织。

  • .o 文件 (Relocatable File)：是 ELF 标准中定义的三种主要类型之一。在 readelf -h 的输出中，它的 Type 字段显示为 REL (Relocatable file)。

  1. .o 文件在 ELF 家族中的位置

ELF 格式标准涵盖了程序生命周期的不同阶段，.o 只是其中一个阶段：

  • 可重定位文件 (.o)：由汇编器生成，包含机器码和数据，但地址未定，不能直接运行。

  • 可执行文件 (Executable)：由链接器将多个 .o 合并而成，包含完整的运行内容（如 a.out），可以直接运行。

  • 共享目标文件 (.so)：动态链接库，在程序运行时被加载。

  1. .o 文件作为 ELF 的具体特征 虽然 .o 是 ELF 格式，但它与最终的可执行文件（EXEC）有几个关键区别：

只有 Section，没有 Segment：

Section指程序节，如.text,.data,.bss等，Segment指程序段常见类型有LOAD等，在下文有对二者的详细描述

.o 文件主要由 Section Header Table 组织，分为 .text、.data、.bss 等逻辑节，方便链接器合并。

它通常不包含 Program Header Table（Segment），因为 Segment 是给操作系统映射内存用的，而 .o 还没到那一步。

包含重定位表 (.rel.text / .rel.data等)：

这是 .o 文件的核心特征。因为它内部引用的函数（如 printf）地址还没确定，它必须带有一张“报修单”，告诉链接器哪里需要填入真正的地址。

基准地址为 0：

在 .o 文件内部，代码段的起始地址通常从 0x00000000 开始。它是一个“相对位置”，只有在链接阶段，链接器才会给它分配真正的“绝对地址”。

ELF 文件整体结构#

ELF文件#

ELF 的三大核心组成：

  • ELF Header（文件头）：位于文件开头，描述文件的总体属性。
  • Section Header Table（节头表）：描述文件中所有 Section 的索引表，是链接视图的地址指南。
  • Sections（节数据区）：文件的实际内容主体，包含指令、数据、符号表等。

ELF Header#

它记录：

  • Magic Number：文件格式标识（7f 45 4c 46）。
  • Type（文件类型）：例如 REL（可重定位文件，即 .o）或 EXEC（可执行文件）。
  • Machine（目标架构）：如 RISC-V、ARM 或 x86-64。
  • Entry point address（入口点地址）：程序执行的起点。在 .o 文件中，此项通常为 0。
  • Start of section headers：节头表在文件中的偏移位置。

关于入口点 (Entry)：.o 文件 e_entry = 0 ，因为 .o 文件尚未分配最终运行地址，且不是完整的执行程序，故无实际入口。

**Section Header Table **：它是 ELF 的“目录”。没有它，链接器就找不到 .text 或 .data 在文件中的具体位置。

Section#

一个 .o 文件通常包含：

Section 作用 备注
.text 编译后的机器指令（代码）。 只读，可执行。
.data 已初始化的全局变量和静态局部变量。 初始值存储在文件中。
.bss 未初始化（或初始化为 0）的全局变量和静态局部变量。 不占用文件空间，仅记录大小。
.rodata 只读常量数据（如字符串字面量、const 变量）。 通常映射到只读内存。
.symtab 符号表，记录函数和变量的定义与引用信息。 链接器解析符号的核心依据。
.strtab 字符串表，存储符号名、节名等文本字符串。 供其他 Section 索引。
.rel.text 针对 .text 的重定位信息。 记录哪些指令地址需要链接时修补。
.rel.data 针对 .data 的重定位信息。 记录哪些指针变量需要链接时修补。

对比.data与.bss#

.bss 段不占文件空间”是一个极其重要的物理特性，它直接体现了二进制文件设计的“吝啬”与高效。

我们可以通过以下三个维度来彻底拆解这个概念：

1.现象：对比 .data 与 .bss#

假设你写了两个全局数组，一个初始化了，一个没初始化：

char data_array[1024 * 1024] = {1}; // 1MB，已初始化，属于 .data
char bss_array[1024 * 1024];        // 1MB，未初始化，属于 .bss

.**data 的情况：**因为你明确要求数组里的第一个值是 1（其余默认是 0），编译器必须在硬盘上的 ELF 文件里实实在在地挖出 1MB 的空间，把这些 1 和 0 存进去。否则，程序运行上电时，从哪里读取这个 1 呢？

结果：你的 .o 或可执行文件体积会增加 1MB。

**.bss 的情况：**C 语言标准规定，未初始化的全局变量默认初始值为 0。既然全是 0，为什么还要在硬盘里存 1MB 的 0 呢？那太浪费空间了。

结果：链接器只在 ELF 文件的 Section Header 里记上一笔：“我需要一个名为 .bss 的空间，大小是 1MB”。但在文件的实际数据区里，不占用任何字节。

2.本质：Size 的两个维度#

在 ELF 的 Program Header 中，有两个关键字段能让你看清真相：

**FileSiz (File Size)：**该段在硬盘文件中实际占用的字节数。

**MemSiz (Memory Size)：**该段在运行内存中申请占用的字节数。

对于包含 .bss 的 Segment 来说： MemSiz 永远大于 FileSiz。 多出来的这部分差额，就是 .bss 的大小。它在硬盘里是“虚”的（不占地方），在内存里是“实”的（占用空间）。

3.谁来负责把“虚”变“实”？#

既然硬盘里没存这些 0，那么程序运行的时候，内存里的 0 是哪来的？

加载器/启动代码的责任：

在 Linux 中：操作系统内核加载 ELF 时，看到 MemSiz > FileSiz，会自动在内存里多分配一块空间，并直接调用清零指令。

在裸机/嵌入式中： 我们的 Startup Code（启动代码）会有一段循环代码，读取链接脚本里导出的 .bss 起始和结束地址，手动用 str 指令把那块 RAM 全部写成 0。

总结#

“不占文件空间”的意思是：

为了节省硬盘/闪存空间，ELF 文件只记录 .bss 段的长度信息，而不存储其具体的零字节内容。

.text / .data：既有长度信息，也有实际内容（代码指令或初始值）。

.bss：只有长度信息，内容全靠程序运行前现场“初始化”。

这就是为什么你可以定义一个巨大的全局数组 char buffer[102410241024]（1GB），而编译出来的可执行文件依然只有几 KB 的原因。

重定位的本质#

重定位到底在修什么？#

回到本质：

当编译器生成机器码时，如果某条指令需要：

  • 访问一个全局变量
  • 调用一个函数
  • 取某个符号地址

但它不知道最终地址是多少

它就会：

在那条指令对应的位置留下一个“待修补记录”

这条记录就进入 .rel.xxx 或 .rela.xxx。

为什么bss，rodata，symtab，strtab没有重定位信息#

为什么 .text 通常有重定位？

因为代码里经常有：

extern int g;
int main() { return g; }

生成的指令里：

lw a0, g

这里的 g 地址未知 → 需要修补 → 产生 .rel.text

为什么 .data 可能有重定位？

比如：

int x = 10;
int *p = &x;

p 的初始值是 &x。

但 x 的地址链接前不知道。

所以：

.data` 里某个位置需要修补 → 产生 `.rel.data

那为什么 .bss 没有重定位？

因为：

.bss 本质上是：

未初始化的全局变量

例如：

int a;
int b[100];

它们：

  • 没有初始值
  • 不占文件空间
  • 链接后只是分配一块零填充内存

为什么 .rodata 通常没有重定位？

.rodata 里一般是：

const char *s = "hello";

字符串常量本身：

"h e l l o \0"

这只是纯字节数据。

它不引用别的符号。

所以：

没有需要修补的地址字段

但是如果：

extern int g;
const int *p = &g;

那么 .rodata 里就会出现重定位。

所以更准确说法是：

.rodata “通常”没有重定位，但不是绝对。

为什么 .symtab 没有重定位？

这个非常关键。

.symtab 是：

符号表

它只是记录：

| 符号名 | 类型 | 所属 section | 相对偏移 |

它不是程序执行内容。

链接器用它来：

  • 解析符号
  • 计算地址
  • 修补 relocation

一旦生成最终 ELF：

  • 可执行文件可能压根不带 .symtab
  • 或者被 strip 掉

所以它本身不需要重定位。

为什么 .strtab 没有重定位？

.strtab 是字符串表。

里面只是：

"main\0"
"printf\0"
"_start\0"

它只是给 .symtab 提供名字。

程序运行时根本不访问它。

没有地址依赖 → 不需要重定位。

总结#

可以这样分类：

Section 是否参与执行 是否可能含未定地址 是否需要重定位
.text 是 经常 是
.data 是 可能 可能
.bss 是 几乎不 很少
.rodata 是 少 少
.symtab 否 否 否
.strtab 否 否 否

更具上述总结重新进行推论#

.symtab

它只是一个符号描述表：

  • 符号名
  • 所属 section
  • 相对偏移
  • 类型

它不参与运行，也不嵌入“待修补地址常量”。

所以：

✔ 永远不会有 .rel.symtab

.strtab

只是字符串池：

"main\0"
"printf\0"

纯文本数据，不涉及地址依赖。

✔ 永远不会有 .rel.strtab

.bss

.bss 的本质是：

“未初始化数据占位区”

它在 .o 文件里甚至不占实际字节，只是一个大小声明。

它内部：

  • 没有初始化值
  • 没有地址常量

因此通常：

✔ 不产生重定位

但注意一个关键点：

如果别的 section 引用了 .bss 里的符号， 重定位出现在“引用者”那里，而不是 .bss 里。

.rodata

通常只是：

常量数字
字符串字面量

如果这些常量不引用其他符号：

✔ 不需要重定位

但如果你写：

extern int g;
const int *p = &g;

那 .rodata 里就嵌入了 &g 的地址占位符。

这时：

✔ 会产生 .rel.rodata

所以它是“可能有”。

.data

如果是：

int x = 10;

没有引用符号：

✔ 不需要重定位

如果是：

extern int g;
int *p = &g;

✔ 需要重定位

.text

最常见需要重定位的 section。

因为：

  • 函数调用
  • 全局变量访问
  • 取地址操作

都会嵌入“未定地址”。

链接发生了什么#

重定位（Relocation）是链接过程的核心步骤。当编译器生成 main.o 时，它并不知道外部函数 foo 的精确地址，因此会在调用处留下一个地址占位符（通常是全 0），并同时在 .rel.text 节中记录一条“修补申请”。

链接器的任务就是：

  1. 计算出 foo 在最终可执行文件中的绝对地址（或相对偏移）。
  2. 根据重定位表的指示，回过头来将这个真实地址填入 main.o 的占位符处。

我们继续使用上文的代码来进行演示

main.c

int foo();
int main() {
  return foo();
}

foo.c

int foo() {
  return 42;
}

然后：

gcc -c main.c
gcc -c foo.c
gcc main.o foo.o -o hello

好，我们现在来使用命令查看各个过程文件的内部信息

(1) 查看可重定位文件 main.o# 

objdump -d main.o

这条命令的意思是：请把 main.o 这个目标文件里的二进制代码，翻译成人能看懂的汇编代码给我看。

输出如下：


main.o：     文件格式 elf64-x86-64


Disassembly of section .text:

0000000000000000 <main>:
   0:	f3 0f 1e fa          	endbr64 
   4:	55                   	push   %rbp
   5:	48 89 e5             	mov    %rsp,%rbp
   8:	b8 00 00 00 00       	mov    $0x0,%eax
   d:	e8 00 00 00 00       	call   12 <main+0x12>
  12:	5d                   	pop    %rbp
  13:	c3                   	ret

(2) 查看重定位表项# 

objdump -r main.o

这条命令的意思是：显示目标文件（.o）中的重定位表（Relocation Table）。


main.o：     文件格式 elf64-x86-64

RELOCATION RECORDS FOR [.text]:
OFFSET           TYPE              VALUE 
000000000000000e R_X86_64_PLT32    foo-0x0000000000000004


RELOCATION RECORDS FOR [.eh_frame]:
OFFSET           TYPE              VALUE 
0000000000000020 R_X86_64_PC32     .text
RELOCATION RECORDS FOR [.text]#

这是你的代码段。它告诉链接器：在 main.o 的机器码里，有些跳转地址是错的，需要修正。

  • OFFSET 000000000000000e: 在 .text 段的第 14 个字节（16 进制的 e）处，有一个需要填空的坑。如果你用 objdump -d 看这一行，你会发现它对应的指令是 call。
  • TYPE R_X86_64_PLT32: 这是一种重定位方式。PLT（Procedure Linkage Table）代表这个函数可能在动态链接库（如 .so 文件）里。它告诉链接器：“请计算从当前位置到目标函数跳转表的相对偏移量。”
  • VALUE foo-0x0000000000000004: 目标是 foo 函数。减去 4 是因为 call 指令本身占用了字节，计算相对偏移量需要从指令结束后的位置算起。 结论： 编译器说：“我不知道 foo 在哪，链接器你帮我把第 e 字节处的 00 00 00 00 换成 foo 的真实偏移量。”
RELOCATION RECORDS FOR [.eh_frame]#

  • .eh_frame 是一个索引表。

    .eh_frame 是一个索引表它的任务是如果程序运行中突然“发生意外”（比如 C++ 抛出异常，或者程序崩溃需要打印调用栈），操作系统需要查找这个“索引表”，搞清楚当前运行到哪一行了、上一级函数是谁、局部变量该怎么销毁。

  • OFFSET 00000020

    在 .eh_frame 这个“索引表”内部的第 32 字节（16 进制的 20）位置，有一个空格。

  • VALUE .text

    这个空格里需要填入 .text 段的起始地址。

  • 为什么要填这个？

    因为“索引表”必须知道它负责的是哪一段代码。 在 .o 文件里，.text 段的地址还没定（是逻辑上的 0），所以 .eh_frame 里的这个“索引项”也只能空着。它在等链接器把 .text 放到内存的具体位置后，再回来更新这个索引指针。

把这两部分输出结合起来看#

.text 里的记录 证明了：你的 main 函数想调用 foo，但目前只有个名字，没有地址。就像你写信填了收件人名字，但没写邮编和地址，邮递员（CPU）没法投递。

.eh_frame 里的记录 证明了：连辅助程序（用于崩溃处理的信息）都还没和代码对齐。

总结#

  • 编译器（Compiler）：负责发现“未定义的引用”并生成重定位表。
  • 链接器（Linker）：负责查阅重定位表，计算真实地址，并执行**“按址抹写”**。

这就是为什么 .o 文件必须经过链接才能运行：没有链接器的修补，程序内部的函数调用和全局变量访问都会指向错误的地址（通常是 0 地址），导致程序崩溃。

第三部分：从 Section 到 Segment#

在上文中，我们拆解了 .o 文件的内部构造。我们看到了 .text、.data、.bss 这些散落在各处的 Section（节），它们是编译器为了方便管理代码和数据而划分的区块。

然而，当我们把多个 .o 文件交给链接器，生成最终的可执行文件时，一个新的问题出现了：操作系统（或加载器）在运行程序时，并不关心你的代码来自哪个 .o 文件，也不关心你细碎的符号表。它只关心最本质的物理需求——这块内存能不能写？那块内存能不能运行？

于是，为了满足“运行”的效率，ELF 文件演化出了第二套视角。这就是我们接下来要讨论的从 Section（链接视角） 到 Segment（运行视角） 。

在最终的 ELF 可执行文件中，Section Header Table（节表）和 Program Header Table（段表/Segment 表）通常是同时存在的。

  • Segment Table (段表/Program Header Table)：
    • 它是给操作系统（Loader）看的。
    • 操作系统运行程序时，根本不关心你的函数名叫什么，它只看 Segment。它会说：“把这一块 40KB 的数据搬到内存地址 $0x400000$，并给它‘只读+执行’权限。”
    • 没有它，程序无法加载运行。
  • Section Table (节表/Section Header Table)：
    • 它是给调试器（GDB）或反汇编工具（objdump）看的。
    • 虽然程序能跑了，但如果你想调试，调试器需要知道：main 函数在哪个位置？全局变量 my_var 在哪？
    • 这些信息依然存在于 Section 表中。
    • 理论上，删掉（strip）Section 表，程序依然能跑，但你就没法调试它了。

Section 是一种逻辑划分单位，它存在的目的是：

  • 组织代码与数据

  • 便于调试

操作系统加载程序时，只做三件事：#

  • 第一步：遍历 Program Header Table (PHT)

    操作系统打开 ELF 文件后，首先读取文件头，定位到 PHT。

    操作系统把 ELF 当成一张“内存置放地图”。PHT 里的每一个条目（Entry）都在说：“我这有一块数据，请帮我安置好。”

    此时，.text、.rodata、.hash 等几十个 Section 已经被链接器“打包”进了几个大的 Segment 里。OS 并不关心里面装的是代码还是常量，它只关心这块二进制的大小。

  • 第二步：找到所有 LOAD

    PHT 里有很多类型的段（比如 PT_INTERP 指向动态链接器），但操作系统最关心的是 LOAD。只有类型为 LOAD 的段才需要被搬进内存。核心参数： 每个 LOAD 段携带了关键的四个参数：

    **File Offset：**这块数据在硬盘文件的哪个位置？

    **Virtual Address ：**这块数据要放在虚拟内存的哪个地址？

    **File Size / Mem Size：**要搬多少字节？（如果 Mem Size > File Size，多出来的部分就是 .bss，OS 直接刷成零）。

    **Flags (权限)：**R (读), W (写), X (执行)。

  • 第三步：按页映射（Memory Mapping）

    这是最关键的性能优化步骤。操作系统并不会真的把文件内容立刻“拷贝”到内存里，而是通过 mmap 系统调用做“映射”。

    对齐分页（Alignment）：由于 CPU 的内存管理单元（MMU）是以“页”（通常是 4KB）为单位工作的，所以每个 PT_LOAD 段的起始地址和长度通常都会对齐到 4096 字节。

    建立映射关系：OS 只是在进程的页表（Page Table）里画了个押：“从虚拟地址A到B的这块区域，对应硬盘文件F的偏移量O。”

    按需加载（Page Fault）：当你真正开始运行第一行指令时，CPU 发现内存里还没数据，触发一个“缺页异常”。此时，OS 才慢吞吞地从硬盘把那一页数据读进物理内存。

为什么它不关心 .text 或 .data？#

这就是权限驱动的设计哲学。

合并同类项： 在 .o 文件里，.text 是执行权限，.rodata 是只读权限。链接器发现它们都“不需要写”，于是把它们合并成一个 LOAD 段，权限设为 R-E。

安全性与效率： 操作系统只需要知道：

这段能跑吗？（执行权限）

这段能改吗？（写入权限）

如果 OS 去关心几百个细小的 Section，内存管理会变得极其碎裂且低效。

Section 如何变成 Segment？#

这是链接器的工作。

我们可以抽象成一句话：

多个 Section 会被合并成一个 Segment。

据个例子，典型映射关系如下：

.text
.rodata
   ↓
合并成一个 RX Segment

.data
.bss
   ↓
合并成一个 RW Segment

也就是说：

Segment 包含 Section 权限
第一个 LOAD .text + .rodata R-X
第二个 LOAD .data + .bss RW-

这是因为：

  • .text 和 .rodata 都是只读
  • .data 和 .bss 都是可写

而现代操作系统以“页”为单位管理权限。它实现了硬件级别的访问隔离，能以极低成本让 CPU 瞬间拦截任何非法指令（如修改代码或执行数据），从而保障系统的稳定与安全。

为什么必须有两套结构？#

这是计算机体系结构设计的精妙之处。之所以在同一个 ELF 文件中既有 Section 又有 Segment，是因为它们分别服务于程序生命周期中两个完全不同的阶段：

1. Section 视角：解决“如何构建程序”（链接阶段）#

Section 是逻辑组织单位。 在链接时，链接器并不关心内存权限，它需要的是对代码和数据进行精确的分类与合并。

  • 核心任务：
    • 符号解析：通过 .symtab 查找函数定义。
    • 重定位：通过 .rel.text 修正指令中的地址。
    • 排布对齐：根据各节定义的对齐要求（Align）在文件中排列数据。
  • 性质：它是编译/链接视图（Linking View），是给链接器看的。

2. Segment 视角：解决“如何运行程序”（加载阶段）#

Segment 是物理映射单位。 在程序运行时，操作系统（或加载器）并不关心代码来自哪个具体的函数，它只关心硬件级的内存属性。

  • 核心任务：
    • 映射虚拟地址：将文件内容一次性加载到指定的虚拟地址空间。
    • 设置页权限：通过 MMU 设置内存页的只读（R）、读写（RW）或执行（X）权限。
    • 资源整合：将几十个零碎的 Section 按权限合并为两三个大的 Segment，以减少内存页浪费（页对齐造成的空间碎片）。
  • 性质：它是执行视图（Execution View），是给操作系统和硬件看的。

3. 结论：语义与物理的解耦#

  • Section 是语义结构：它代表了代码和数据的“逻辑属性”（如：这是已初始化的数据，那是重定位信息）。
  • Segment 是物理映射结构：它代表了代码和数据的“硬件属性”（如：这 4KB 内存必须设为只读且可执行）。

总结： 如果只有 Section，加载程序时会产生大量的碎片化内存映射，严重影响性能；如果只有 Segment，链接器将失去精确控制代码合并与地址修正的能力。两套结构的并存，实现了开发阶段的灵活性与运行阶段的高效性之间的平衡。

硬件视角下的 Segment 加载与执行#

当操作系统加载程序时：

  1. 为每个 LOAD 创建虚拟内存区域
  2. 分配页表
  3. 设置页权限（R/W/X）
  4. 将文件内容映射到页

CPU 执行时发生的事情：

PC → 虚拟地址
  ↓
MMU 查页表
  ↓
物理地址
  ↓
取指

CPU 并不知道：

  • 哪些字节来自 .text
  • 哪些来自 .rodata

它只知道：

当前页是否可执行

一个重要现象：.bss 不存在于文件中#

在 Program Header 中你会看到：

FileSiz < MemSiz

这意味着：

  • 文件里没有 .bss 的内容
  • 但运行时会分配空间

加载器会：

  • 分配额外内存
  • 自动清零

它在文件中没有实体，但在 Segment 中有空间。

从体系结构角度的真正结论#

现在我们可以总结：

ELF 同时服务于两个阶段#
阶段 结构 关注点
链接阶段 Section 逻辑组织
加载阶段 Segment 地址映射

而 CPU 只认：

虚拟地址 + 页权限

第四部分：从 ELF 到 bin：回归物理真实#

为什么需要从 ELF 转换为 bin？#

  • ELF 的冗余性：ELF 文件包含大量调试信息（Debug Info）、符号表（Symbol Table）和节头表（Section Header Table）。这些信息对链接器和操作系统有用，但对于裸机硬件（如 Flash 存储器）来说是多余的。
  • 硬件的直接性：嵌入式 CPU 上电后，通常从固定物理地址读取原始机器码，它无法解析 ELF 复杂的段结构。

转换的本质：提取 LOAD 段#

使用 objcopy 工具（如 arm-none-eabi-objcopy）执行转换：

  • 动作：丢弃所有非 LOAD 类型的 Segment。
  • 动作：丢弃所有 Section Header 和 Program Header。
  • 结果：只保留 PT_LOAD 段中的原始数据，并按照其在物理内存中的相对位置进行排列。

关键差异对比#

特性 ELF 文件 Bin 文件
内容 代码+数据+元数据（符号、索引） 纯粹的代码和数据镜像
结构 结构化存储，支持重定位和动态解析 扁平化的二进制流，与物理地址一一对应
大小 较大（包含大量辅助信息） 较小（仅包含运行所需的有效载荷）
用途 调试、链接、操作系统加载 直接烧录到 Flash 运行

转换后的物理布局#

在 .bin 文件中，原本分散在 ELF 不同位置的 LOAD 段会被拼接在一起。若两个段之间在物理地址上不连续，objcopy 通常会用 0x00 或 0xFF 填充这些空隙，以保持物理地址的线性关系。
